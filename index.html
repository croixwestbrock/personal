Anomaly Detection Research - Research Experience for Undergraduates: ARA Project

Through Research Experience for Undergraduates at Iowa State, I was able to gain the opportunity to work for the WiCI Research Center's ARA project during the Spring 2024 semester. During this time, I researched anomaly detection in time series weather data under my mentor, using data collected from two Iowa State locations: the Agronomy Farm and Wilson Hall. Some Python packages which I utilized included pandas, numpy, matplotlib, and sklearn. I began by cleaning and preprocessing the data, resulting in cutting the Wilson Hall and Agronomy Farm data frames from containing 26 and 30 columns respectively down to the 8 and 9 most important columns in explaining variance. More specifically, this was done through use of PCA and visual analysis of correlations and various graphs created. I finished preprocessing by filled any missing values with mean values and scaling and normalizing both data sets. After having the data cleaned and ready for testing, I implemented VAR, IsolationForest, and DBSCAN models, then graphically displayed the returned anomalous data points on time series graphs of each of the location's columns to gain insight on what columns seemed to contibute heavily, randomly, or not much at all, to determining a point as anomolous.

After testing those models, I began experimenting with the ARIMA model in various ways. I first began by detemining what columns of each data set would be best for testing this model, as ARIMA is a univariate model and hence can only take one column of the data at a time. To determine this, I looked for the columns which had a visibly normal range which spiked occassionally. I chose 3 columns from each data set to put through ARIMA models. The way I went about these ARIMA models is significantly different from the previous models, however, in the sense of the train test split.

For these ARIMA models, I ran the model on each column for 4 different train test splits. The first was a simple random 20-80 train test split as a sort of base assessment. Second was a DBSCAN clustering split, with the training data being all the records which got put into some sort of cluster, which was around 10% of the total data, and all unclustered data being the test data. Next, we used an OPTICS clustering split in the same way as we used DBSCAN, with OPTICS producing 18% clustered data for our training split. Finally, we used K-Means clustering for our final split, where we K-Means clustered the data into 3 clusters before manually inspecting each cluster to determine which of the clusters contained most of the average data points, making that the training data, and making the clusters with mostly spiked data our testing data set. For each split of each column, we displayed two graphs showing what values were determined by the ARIMA model to be anomalous versus normal, as an example is shown to the right.

Movie Rating Predictor

In this project, me and two classmates aimed to determine if you can accurately predict the rating of a video game based off of its charateristics. We began this by exploring the dataset we chose to work with and determine what attributes of a game would actually be useful to predicting its rating. We began with the release year, certificate, plot, title, number of rating votes, the rating itself, and 9 different game genres, each of boolean type for each game, where a game could have multiple genres. From here, we first removed unncessary columns, any duplicates, and records with missing values, and any game which had less than 100 votes on it. This brought our dataset from 20,800 records down to 3,315 records, and leaving the columns at rating, certificate, and all the different genres. All the genres were changed from true/false to 1/0, and we used one-hot encoding to create columns for each certificate option which could be represented by a 1 or a 0. Now that we had all our data besides the actual ratings into simple 1s and 0s, we began implementing different classifiers.

The classifiers we used are shown to the right, and with these classifiers we used a stratified KFold with 2 splits to split the data due to the high variance in the number of games that each genre had, for example action and adventure had significantly more games than other genres. We then used the accuracy_score function from sklearn to determine the accuracy of each classifier compared to the real ratings. The results we found were that the Logistic Regression classifier worked the best of the classifiers, however is still only had a 45.7% accuracy rating. When compared to a random 1-10 guess which yields a 10% predicted accuracy, our classifiers and predictions were still significantly better than guessing, however due to its shortcomings with straight forward accuracy, we determined our project would not be able to hold up in real world implementation.

Solving Rush Hour

The objective of this project was to take the common children's game Rush Hour and write algorithms based around winning the game in the shortest number of moves. This work was set up having the main class, GameBoard, along with three helper classes: Car, Pair, and HashKey. Car objects consisted of a car's number, whether it moves horizontal or vertical, and what grid spots it occupies. Pair defines moves, formatted as pairs of car id and the direction it moves: n, s, e, or w. HashKey is an altered HashKey to be used for Car objects. The algorithms discussed are all implemented in GameBoard. The setup of the board is given through files, which in GameBoard are read to define all the board's starting features. Files are formatted as the first line telling how many cars there are, followed by that number of additional lines, each defining a car's grid spots. The first of these cars' lines is for the car wanting to get out.

The first of the large algorithms implemented was to get the plan for the shortest path of moves which wins the game. This was done using a modified version of Dijkstra's Algorithm while tracking the path along the way. A list of Pair objects showing the path followed is returned.

The second large algorithm writen was written utilizing breadth-first-search logic to find the different number of paths that can solve the board in the smallest solvable number of steps. To do this, the algorithm takes the path which first solve the board, then mark the layer of that search as the shortest path layer, or the smallest amount of steps that the board is solvable in. The other paths are then followed through until they are either solved in that same number of steps and added to the count, or until they require more steps and are disregared. The count of paths which solved the board in the minimum move amount is returned.

The final large algorithm, which was used as a helper method in the two main algorithms just described, was getting all the next possible board states of a given board. To do this, the method went through all the cars and determined whether it could move up, down, left, or right. For each possible move, a new GameBoard object was added to an ArrayList. Each GameBoard in this list only consists of one car moving one position from the starting board. After all moves for all cars are considered, that ArrayList is returned.

Pacman Logic

This project was used to work with and utilize inheritance in order to code the behavioral logic for PacMan each of the ghosts. To do this, I created an AbstractActor class which would be the overall parent class overseeing all characters. The next layer of inheritance consists of PacMan and BasicGhost, both child classes of AbstractActor. The PacMan class was used to specify its specific movements in terms of turning, along with updating his location on the maze. BasicGhost defines the ways all the ghosts would move, in terms of getting and setting its mode: frightened, dead, or normal. This class also calculates the next cell that the ghost should go to based its mode, and if it is in chase mode, which ghost it is, as each ghost has a unique target calculation for catching PacMan while in chase mode. Finally, classes for each of the four ghosts were created as child classes of BasicGhost, which simply consisted of constructors in order to simply classify which ghost is which.

Lines Game

This project aimed to make a line connection game, where given a grid of line start and end points, one must connect all the lines without illegally crossing over any other lines. The logic for this was done using two Java classes, Util and LinesGame. Util holds features for setup such as reading a file with game descriptors in it and turning them into a list of LinesGame objects, creating the base line objects that the game begins with, and checks for whether there is a line segment or illegally crossing lines present. The LinesGame class works with individual games, holding the alterations of lines, tracking the number of moves, and checking if the user has won.

Golf Driver Shopping Catalog

In this project, a classmate and I worked to create a webpage which simulates a shopping catalog for golf drivers. We designed this basing from a MongoDB database which homed all the information used throughout our webpage on all the 9 drivers we used. That database connected to a Node.js backend which pulled the data into a server, which we then would use hooks in our frontend to access that information. We implemented a React frontend which used these hooks to get all the data needed and display our webpage. In the webpage itself, we used a single page design, making buttons change variables to show and hide different sections rather than opening new webpages. All aspects of our webpage is one single site. We also used Bootstrap and other CSS for visual design and appeal.

The different views we had began with the catalog view, which had a card for each driver, displaying image, name, price, rating, and option to add or remove from your cart. If a user clicks on one of these cards, the catalog view is hidden, and the full information for that specific driver is shown, which sees more specific information on product options for the driver itself, as well as design aspects and information to do with how the driver is made and what it helps with. After a user adds what they want to their cart, they can hit the check out button, which will hide everything else and pull up the checkout screen, where users can review what they have in their cart, and enter their payment information. Every input in payment information also has a validity check to ensure the user did not forget to enter anything or enter anything in an incorrect format, such as email not containing an @ symbol. If all these pass their checks when the user clicks submit, the order is placed.

Our website does not retain any information actually entered, nor does it actually place any real orders or charge any actual cards, for obvious reasons.

From there, the previous part is hidden and the user is shown an order summary containing all the information they entered, and the card number having X for all but the last 4 digits. The return button hides this summary and return the user to a cart view where the cart is reset to not having anything in it. In our navigation bar at the top of our page, there are 3 options: "View Catalog" and "Golf Driver Catalog" both take the user back to the catalog view with everything else hidden without resetting the user's cart, and "About" takes the user to a simple page showing myself and my classmate's information, along with the information of the professor and the class we coded this in.
